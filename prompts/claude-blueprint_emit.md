# Task: Lean Blueprint Emission

## Objective

Emit a Lean 4 benchmark blueprint file and companion sidecar artifacts from `outline.json`. The
blueprint must follow the upstream `autoformalization_pilot` NODE block style exactly. Benchmark
theorem nodes must remain `:= by sorry`. This is Stage 7 of the benchmark outline pipeline.

## Inputs

- Problem ID: `{problem_id}`
- Outline file: `{outline_file}`
- Mathlib map file: `{mathlib_map_file}`
- Assumption profile file: `{assumption_profile_file}`
- Selected profile: `{selected_profile}`
- Benchmark output directory: `{benchmark_dir}`
- Lean project root: `{lean_project_root}`

## What To Do

### Step 1: Read all inputs

Read `{outline_file}`, `{mathlib_map_file}`, and `{assumption_profile_file}`.
The assumption profile is human-authored ŌĆö its `open_nodes` and `assumed_nodes`
reflect explicit human choices, not AI-generated defaults.

### Step 2: Determine imports

Based on the nodes in the outline:
- Always include `import Mathlib.Tactic`.
- Add specific mathlib imports for any node classified `existing` in the mathlib map, if a specific
  module path is known.
- Keep imports minimal but sufficient.

### Step 3: Emit `problem_blueprint.lean`

Write `{benchmark_dir}/blueprint/problem_blueprint.lean`.

The file must follow this structure exactly:

```
-- Generated by latex_to_lean_question pipeline

import Mathlib.Tactic
<additional imports>

/-! NODE
  \\name: NodeName
  \\inputs: ["dep1", "dep2"]
  \\type: definition|theorem|hypothesis
  \\natural: <natural language statement>
  \\NL_proof: <natural language proof sketch, or empty for non-theorems>
-/
<Lean 4 declaration>

/-! NODE
  ...
-/
<Lean 4 declaration>
```

Rules for each NODE block:

1. The comment block `/-! NODE ... -/` must be immediately followed by the Lean declaration.
   Only blank lines are allowed between `-/` and the declaration keyword.
2. Use `\\name:`, `\\inputs:`, `\\type:`, `\\natural:`, `\\NL_proof:` ŌĆö exact field names,
   no variants.
3. `\\inputs:` must be a JSON-style list: `["NodeA", "NodeB"]` or `[]`.
4. For `theorem` type nodes: emit `theorem NodeName ... : ... := by sorry`.
5. For `hypothesis` type nodes: emit `def NodeName : Prop := ...` or the Prop-valued expression.
6. For `definition` type nodes: emit `def NodeName ...` or `structure NodeName ...`.
7. `\\NL_proof:` must be empty (blank after the colon) for `definition` and `hypothesis` nodes.
8. The `formal_stub` from `outline.json` is your starting point for the Lean declaration. Refine
   it to be syntactically valid Lean 4. Use `loogle_search` to verify any mathlib identifiers.
9. Benchmark `theorem` nodes (those in `open_nodes` for the selected profile) must end with
   `:= by sorry`. Do not replace any of these `sorry`s.
10. Node order must respect dependencies (each node's inputs appear earlier in the file).

### Step 4: Emit `problem_blueprint.json`

Parse the Lean file you just wrote and emit `{benchmark_dir}/blueprint/problem_blueprint.json`
with the same structure the upstream parser produces:

```json
[
  {{
    "name": "NodeName",
    "inputs": ["dep1", "dep2"],
    "type": "theorem|definition|hypothesis",
    "natural": "...",
    "formal": "<full Lean declaration text>",
    "NL_proof": "..."
  }}
]
```

### Step 5: Emit `problem_graph.mmd`

Write `{benchmark_dir}/blueprint/problem_graph.mmd` as a Mermaid diagram:

```
graph TD
    NodeA["NodeA (definition)"]
    NodeB["NodeB (hypothesis)"]
    NodeC["NodeC (theorem)"]
    NodeA --> NodeC
    NodeB --> NodeC
```

Use the node name as label. Add the node type in parentheses. Draw an arrow for each dependency in
`inputs`.

### Step 6: Verify build

Run `lake build` in `{lean_project_root}` to confirm the blueprint compiles (with `sorry` allowed).
If the build fails, diagnose and fix the declaration syntax. Do NOT fix failures by removing sorry
or attempting proofs.

## Hard Rules

1. Benchmark theorem nodes (those in `open_nodes`) must remain `:= by sorry`.
2. Do not replace any `sorry` with a tactic proof.
3. Do not attempt proof search.
4. Use exactly the field names `\\name:`, `\\inputs:`, `\\type:`, `\\natural:`, `\\NL_proof:`.
   Do NOT use `\\informal:` or any other variant.
5. The comment block and its Lean declaration must be adjacent (only blank lines between `-/` and
   declaration keyword).
6. The `\\inputs:` value must be a valid JSON array literal.
7. `hypothesis` nodes are `def NodeName : Prop := ...`, not `theorem`.
8. The Lean file must build with `lake build` and `sorry` present.

## When Done

Print a brief summary:
- File paths of the three artifacts emitted.
- Total node count.
- Number of `sorry` occurrences in the blueprint (should equal the number of open theorem nodes).
- Build result (pass/fail).
- Any syntax fixes applied.
